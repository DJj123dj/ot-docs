## üíæ System Requirements \{#requirements}
If you're running Open Ticket in a large server, **shared hosting or low-end VPS setups** may not be enough. Here's what you should consider:  

- **CPU & RAM:** Open Ticket is lightweight, but make sure you have at least **1-2 CPU cores** and **1GB+ RAM**.
- **Disk Type:** A **SSD or NVMe drive** is recommended, especially if you‚Äôre using a local (`json` or `sqlite`) database.  
- **Uptime & Stability:** Consider using a **dedicated VPS** with good uptime.

## üóÑÔ∏è SQLite Database \{#database}
By default, Open Ticket **stores data in JSON files**.
While this works for small servers, it can **slow down** with thousands of tickets.
Switching to an <C color="blue">**SQLite database**</C> improves **speed and reliability**.  

For most servers, **SQLite** is a great alternative to JSON. It‚Äôs lightweight and doesn‚Äôt need extra setup. 
- **‚úÖ Faster ticket loading and saving**  
- **‚úÖ Faster command responses**  
- **‚úÖ Reduces file corruption risks**  
- **‚úÖ No external database server needed**

## üõ†Ô∏è Optimizing Plugins & Features  
Not all features are **needed for every server**. Disabling unnecessary options reduces **CPU & memory usage**.  

#### üîß Reduce Heavy Features  
- <C color="red">**Disable Unused Commands:**</C> If you don‚Äôt use certain commands, **disable them** in the settings.  
- <C color="red">**Limit Active Options:**</C> Too many different options will slow down the bot.
- <C color="red">**Limit Active Panels:**</C> If you have **many ticket panels**, only keep the essential ones.  
- <C color="red">**Disable Logs:**</C> If the ticket logs slow down the bot a lot, consider turning them off.

#### üéØ Set Limits
- <C color="green">**Enable Global Limits:**</C> Only one ticket per user.
- <C color="green">**Enable Ticket Limits:**</C> Only one ticket per user per ticket/option type.
- <C color="green">**Enable Ticket Cooldowns:**</C> Users can create a ticket and have to wait an hour to create the next one.
